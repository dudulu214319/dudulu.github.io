秒杀项目总结

2024/3/2 项目 项目笔记

秒杀项目复盘总结

  • 难点：0/1处理，内存（EmptyStockMap.put(goodsId, true)）/redis标记（），重复抢购（建立唯一索引？和redis对象缓存）
  • 最核心的东西就是对【大量用户同时点击秒杀按钮发起访问】的处理
    • 页面优化（缓存，静态化）：减少用户到服务端来“读”数据，或者让他们读更少的数据
    • 服务优化：并发写——lua脚本原子化库存预减，redis分布式锁，服务端秒杀事务原子化，MQ流量削峰，客户端异步轮询
    • 安全优化：验证码，路径隐藏，拦截器interceptor访问限流与AugumentResolver通过UserContext交互。

为什么用消息队列？为什么选用RabbitMQ而不是Kafka/RocketMQ?

  • 解耦（A系统RPC调用BCD新增E系统的接口？）

  • 异步

  • 削峰（OrderService.seckill(user, goodsVo)与MySQL交互，写在receiver里），大不了就海量积压嘛，MySQL按照自己的上限性能慢慢处理就好。

  • 缺点
    • 可用性降低（MQ挂了也就用不了了）
    • 复杂度提高（需要解决顺序性、可靠性、消息丢失、重复消费等一系列问题）
    • 一致性问题——异步消费者未必能”正确“处理数据，需要做专门的库存检测

如何解决RabbitMQ消息丢失的问题？

  • 生产者丢失（网络波动），可以使用事务机制/异步confirm机制（MQ回复ack，nack）
  • queue丢失，设置持久化
  • 消费者提前ACK，但处理消息（验证库存，重复和创建订单）时可能寄，可以改成手动提交ACK

限流算法

  • 令牌桶

    • 令牌桶的好处之一就是可以方便地应对 突发出口流量（后端能力的提升，最适合秒杀）。
    • 令牌桶的大小固定，令牌的产生速度固定（更上一层作了限流），但是消耗令牌（即请求）速度不固定（可以应对一些某些时间请求过多的情况）；每个请求都会从令牌桶中取出令牌，如果没有令牌则丢弃该次请求（消耗比发放快 ）。

  • 漏桶

    • 漏桶大小固定，处理速度固定，但请求进入速度不固定（在突发情况请求过多时，会丢弃过多的请求）

  • 计数器

    • 在一段时间间隔内（时间窗/时间区间），处理请求的最大数量固定，超过部分不做处理。（直接报错）

Spring相关

@Configuration

  • @Configuration注解标识的类中声明了1个或者多个@Bean方法，Spring容器可以使用这些方法来注入Bean。
  • 这个方法就向Spring容器注入了一个类型是MyBean名字是myBean的Bean
@Configuration
public class AppConfig {
  //这个方法就向Spring容器注入了一个类型是MyBean名字是myBean的Bean
  @Bean
  public MyBean myBean() {
     // instantiate, configure and return bean ...
  }
}

@Transactional

  • 加在seckill方法上
  • @Transactional 注解也可以添加到类级别上。当把@Transactional 注解放在类级别时，表示所有该类的公共方法都配置相同的事务属性信息
  • 但是如果对 Spring 中的 @transaction 注解的事务管理理解的不够透彻，就很容易出现错误，比如事务应该回滚（rollback）而没有回滚事务的问题。
  • 分布式事务：将减库存与生成订单操作组合为一个事务。要么一起成功，要么一起失败。

@Component

  • 表示一个带注释的类是一个“组件”，成为Spring管理的Bean。当使用基于注解的配置和类路径扫描时，这些类被视为自动检测的候选对象。

HandlerInterceptor

  • preHandle函数

分布式锁（lua脚本、redis事务）

  • 在多线程环境中，如果多个线程同时访问共享资源（例如商品库存、订单），会发生数据竞争，可能会导致出现脏数据或者系统问题，威胁到程序的正常运行。
  • 区分于本地锁

  • 我们需要用到分布式锁，这样的话，即使多个线程不在同一个 JVM 进程中也能获取到同一把锁，进而实现共享资源的互斥访问。

  • 秒杀中实际使用的是lua脚本，在redis 的官方文档中有描述lua脚本在执行的时候具有排他性，不允许其他命令或者脚本执行，类似于事务。是存在的另一个问题是，它在执行的过程中如果一个命令报错不会回滚已执行的命令，所以要保证lua脚本的正确性。
  • call在执行命令的时候如果报错，会抛出reisd错误终端执行，可以稍微比普通事务好一点。

限流、削峰部分的设计

  • 采取发放令牌机制（控制流量），根据商品id和一串uuid产生一个令牌存入redis中同时引入了秒杀大闸，目的是流量控制，比如当前活动商品只有100件，我们就发放500个令牌，秒杀前会先发放令牌，令牌发放完则把后来的用户挡在这一层之外，控制了流量。
  • 高并发流量的限制（泄洪）：利用线程池技术，维护一个具有固定线程数的线程池。设置最大线程数来限制浪涌流量。每次只放固定多用户访问服务，其他用户排队。另外一种实现方式就是J.U.C包中的信号量（Semaphore）机制。可以有效的限制线程的进入。
  • 异步下单

用户登录的问题（分布式会话）

  • 做完了分布式扩展之后，发现有时候已经登录过了但是系统仍然会提示去登录，后来经过查资料发现是cookie和session的问题。然后通过设置cookie跨域分享以及利用redis存储token信息得以解决。

线程池技术中核心线程数的取值有经验值吗？

  • CPU密集型业务：N+1

  • IO密集型业务：2N+1

为什么用Redis做秒杀缓存？

  • 高性能（内存快）
  • 高并发（单台设备的 Redis 的 QPS 是 MySQL 的 10 倍，Redis 单机的 QPS 能轻松破 10w，而 MySQL 单机的 QPS 很难破 1w），应该是依赖于epooll多路复用
  • 丰富的数据结构（跳表啥的）
  • 即时终止
  • 方便异步削峰
  • binlog异步更新缓存
